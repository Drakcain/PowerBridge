# QR / Import Contract

PowerBridge uses a local setup QR contract to transfer basic PC network setup data from PowerBridge Windows Companion to the PowerBridge Android app.

The QR contract is setup-only. It is not a wake command channel, relay command channel, authentication channel, secret transfer mechanism, cloud pairing token, or remote-control protocol.

## Current Schema

```text
powerbridge.local_setup.v1
```

This schema is generated by PowerBridge Windows Companion and accepted by Android import.

## Purpose

The contract prepares or updates an Android `Local Wi-Fi Wake` profile.

It allows Android to receive the local network values needed for Wake-on-LAN without requiring the user to manually type every field.

The Android app should always preview the imported setup before saving it.

## Current Role in PowerBridge

The QR import flow currently supports setup for:

* `Local Wi-Fi Wake`

The same non-secret PC setup values may also be reused later by user-owned home-side relay flows, such as:

* `Old Android Device`
* `My Own Server / Home Relay Server`

The QR contract should not silently activate a relay method. Relay setup must remain explicit and user-reviewed.

## Generated Fields

| Field          | Purpose                                                  |
| -------------- | -------------------------------------------------------- |
| `schema`       | Identifies the setup contract version                    |
| `pcName`       | Suggested profile or PC name                             |
| `targetMac`    | Target PC MAC address used for Wake-on-LAN               |
| `targetIp`     | Target PC IPv4 address at detection time                 |
| `gateway`      | Detected default gateway                                 |
| `subnetPrefix` | Detected subnet prefix length                            |
| `broadcastIp`  | Detected local broadcast IP                              |
| `adapterName`  | Detected Windows adapter label                           |
| `adapterType`  | User-facing adapter type, such as wired or wireless      |
| `source`       | Indicates the payload was generated by Windows Companion |
| `createdAt`    | Timestamp for when the payload was generated             |

## Field Safety

The generated fields are not passwords or authentication secrets, but they may still reveal local network details.

Treat generated QR payloads and setup JSON files as machine-local setup data because they may contain:

* PC names
* MAC addresses
* local IP addresses
* gateways
* broadcast addresses
* adapter names
* generated timestamps
* local setup structure

Do not publish real generated payloads unless they are redacted.

## Forbidden Fields

The setup QR must never include credentials, relay secrets, account data, private keys, or shared infrastructure secrets.

Forbidden fields include:

| Field           |
| --------------- |
| `username`      |
| `password`      |
| `adminPassword` |
| `relayToken`    |
| `token`         |
| `apiKey`        |
| `secret`        |
| `credential`    |
| `privateKey`    |
| `sshKey`        |
| `winrmPassword` |
| `rdpPassword`   |
| `cloudToken`    |
| `refreshToken`  |
| `accessToken`   |
| `bearerToken`   |
| `clientSecret`  |
| `storePassword` |
| `keyPassword`   |

Android import should reject payloads that include forbidden fields.

## Import Behavior

Android accepts the payload through:

```text
Settings -> PC Profiles -> Scan Setup QR
```

Expected Android behavior:

1. Scan the QR code.
2. Parse the `powerbridge.local_setup.v1` JSON.
3. Reject malformed payloads.
4. Reject unsupported schemas.
5. Reject payloads with forbidden fields.
6. Validate required Wake-on-LAN setup fields.
7. Show an import preview.
8. Let the user choose `Create New Profile` or `Update Current Profile`.
9. Save the setup as a `Local Wi-Fi Wake` profile unless the user explicitly chooses another supported setup path.
10. Keep relay tokens, passwords, credentials, API keys, and secrets out of the import path.

The QR import flow should not silently activate a relay method.

## Import Preview

Before saving, Android should show a reviewed import preview.

Recommended preview fields:

* source
* created time
* PC name
* adapter type
* masked MAC address
* IPv4 address
* gateway
* broadcast IP
* wake port, if present in a future non-secret schema
* target method or intended setup type

The preview should avoid showing full sensitive-looking values where masking is practical.

## Windows Companion Behavior

PowerBridge Windows Companion generates the QR from the same JSON payload used by CLI output. There should be one import contract, not separate GUI-only, CLI-only, or QR-only schemas.

Expected Windows Companion behavior:

1. User selects a Windows adapter.
2. User clicks `Detect This PC`.
3. Companion detects local network values.
4. User clicks `Generate QR Code`.
5. Companion renders a scan-safe QR code from the `powerbridge.local_setup.v1` JSON.
6. Android scans and imports the payload.

The primary QR should remain scan-safe:

* black-on-white
* sufficient quiet zone
* readable by standard Android camera/scanner behavior
* no stylized color mode as the default scan target
* no branding overlay that interferes with scanning

## GUI Output Behavior

The Windows Companion GUI displays the QR on screen for scanning.

The simplified GUI pairing flow should keep the QR session-local unless an explicit export feature is intentionally added later.

Current expected GUI behavior:

* display QR for the current session
* do not require clipboard use
* do not require admin rights
* do not expose relay tokens
* do not expose credentials
* do not act as a wake command sender

CLI mode may generate JSON or QR output for testing or troubleshooting. Generated files may contain real local machine values and should be treated as machine-local setup data.

Generated CLI output should stay under an ignored output folder, such as:

```text
windows-companion/output/
```

## Security Rules

The QR contract must remain secret-free.

The QR must not include:

* relay tokens
* passwords
* account credentials
* API keys
* private keys
* Windows login data
* remote desktop credentials
* WinRM credentials
* SSH credentials
* cloud service credentials
* signing material
* keystore passwords
* shared relay credentials
* developer-managed infrastructure values

The QR should contain only local setup values needed to prepare a Wake-on-LAN profile.

## Public-Safety Rules

Generated setup payloads may contain real local network values, including MAC addresses and private IP addresses.

Do not publish generated QR payloads, generated QR screenshots, generated JSON files, logs, screenshots, or diagnostics containing real setup values unless intentionally redacted.

Public examples should use placeholders only.

Safe example values:

```json
{
  "schema": "powerbridge.local_setup.v1",
  "pcName": "Gaming PC",
  "targetMac": "AA:BB:CC:DD:EE:FF",
  "targetIp": "192.168.1.100",
  "gateway": "192.168.1.1",
  "subnetPrefix": 24,
  "broadcastIp": "192.168.1.255",
  "adapterName": "Ethernet",
  "adapterType": "wired",
  "source": "windows_companion",
  "createdAt": "2026-01-01T12:00:00Z"
}
```

## Non-Goals

The QR contract is not:

* a wake command
* a boot command
* a relay command
* a remote-control channel
* a credential exchange
* a cloud pairing token
* a Windows agent protocol
* a smart-home protocol
* a smart plug protocol
* a router configuration protocol
* a VPN setup protocol
* a remote desktop setup flow
* a shutdown or restart command path

## Removed Method Guardrail

The QR contract should not reintroduce removed PowerBridge method families.

Do not add fields or behavior for:

* TV / streaming relay
* Fire TV / Android TV relay
* Roku relay
* smart plug / hardware power-cycle boot
* router / VPN as a standalone method
* shutdown / restart workflows

If future docs mention those concepts, they should appear only as removed or out-of-scope context.

## Compatibility Rule

Future versions may add optional non-secret fields, but they must not break `powerbridge.local_setup.v1` import compatibility.

If a future schema is needed, it should be versioned explicitly instead of changing the meaning of existing fields silently.

Example future schema naming:

```text
powerbridge.local_setup.v2
```

Until then, QR contents should stay plain `powerbridge.local_setup.v1` JSON.

## Future Extension Rules

Future optional fields must be:

* non-secret
* user-reviewable
* backward-compatible where possible
* safe to display in an import preview
* safe to store in a local PC profile
* documented before public release

Future optional fields must not include:

* credentials
* tokens
* cloud secrets
* signing material
* private keys
* remote command instructions
* hidden method activation flags
* developer-managed infrastructure defaults

## Added Recommendations

Before changing the QR contract:

* confirm Android import still rejects forbidden fields
* confirm Android import still previews before saving
* confirm Windows Companion and Android use the same schema
* confirm generated QR output remains scan-safe black-on-white
* confirm public examples use placeholders only
* confirm generated output remains ignored by Git
* confirm no relay method is silently activated
* confirm removed methods are not reintroduced through new fields
* create `powerbridge.local_setup.v2` only if compatibility cannot be preserved safely
